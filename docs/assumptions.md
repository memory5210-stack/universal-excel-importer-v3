# 需求理解与假设说明

## 概述

本考试题目中有多处业务规则未给出精确数值或完整定义，以下是我对这些"留白点"的假设、设定依据及设计说明。

---

## ① 分级审批金额阈值

| 参数 | 设定值 | 依据 |
|------|--------|------|
| 一级审批金额上限 | 500 元 | 参考中小物流企业日常理赔单均金额（300-800 元），取中下值作为一级审批上限，降低低风险工单的审批层级 |
| 二级审批金额下限 | 5000 元 | 大额赔付通常涉及保险理赔或管理层决策，5000 元作为分水岭可使约 80% 的工单在一级完成、20% 需要二级审批，比例合理 |

实现为 `ConfigRule` 表中可配置的 `approval_amount_threshold_l1` 和 `approval_amount_threshold_l2`，支持后台动态调整。

## ② 审批超时时长

| 参数 | 设定值 | 依据 |
|------|--------|------|
| 审批超时时长 | 120 分钟 | 物流行业工单处理时效 SLA 通常为 2-4 小时，取 2 小时作为超时阈值，平衡"给审批人足够时间处理"和"不让工单积压"两个目标 |

超时后处理策略：
- 待审批/一级审批中超时 → 自动升级到二级审批（异常需要更高关注度）
- 二级审批中超时 → 自动驳回（二级审批通常涉及更高金额，超时不处理视为放弃）

## ③ 重提次数上限

| 参数 | 设定值 | 依据 |
|------|--------|------|
| 最大驳回次数 | 3 次 | 三次驳回后自动关闭的规则参考了物流行业通用的"三次机会"惯例。第 1-2 次驳回后可以补充资料重新提交，第 3 次仍不符合要求则自动关闭工单，避免无限循环 |

## ④ 物流异常类型映射（各类型对应的下游动作）

| 异常类型 | 下游执行动作 | 涉及赔付 | 涉及库存 |
|----------|-------------|---------|---------|
| 丢件 (lost_delivery) | 理赔 + 重新发货 | 是（方向：赔付客户） | 是（重新发货扣减库存） |
| 破损 (damaged) | 理赔 + 退货入库 | 是（方向：赔付客户） | 是（退货入库增加库存） |
| 客户拒收 (refused) | 退货入库 + 重新入库 | 否 | 是（退货入库增加库存） |
| 超时未签收 (timeout) | 重新发货（联系客户确认后） | 否 | 是（重新发货扣减库存） |
| 收货地址错误 (wrong_address) | 重新发货（更新地址后） | 否 | 是（重新发货扣减库存） |

设计考量：
- 涉及货损的异常（丢件、破损）同时产生赔付和库存变动
- 仅涉及服务流程的异常（拒收、超时、地址错误）只影响库存，不产生赔付
- 品控类异常（数量不符、外观破损等）赔付方向为"向供应商追偿"，与物流类异常的"赔付客户"严格区分

## ⑤ 角色权限划分

| 角色 | 权限范围 |
|------|---------|
| admin | 系统管理：用户管理、规则配置、查看所有工单和日志 |
| reporter（操作员） | 扫描录入、手工上报异常、查看自己上报的工单 |
| qc_supervisor（品控主管） | 品控误判复核快速放行、查看品控相关工单、配置品控规则 |
| approver_l1（一级审批人） | 审批一级工单（金额 ≤ 500 元）、查看分配给自己的工单 |
| approver_l2（二级审批人） | 审批二级工单（金额 > 500 元）、超时升级工单 |

权限设计原则：
- **自批自核禁止**：上报人不能审批自己提交的工单
- **最小权限**：一级审批人不能审批二级工单，反之亦然
- **品控主管与审批人分离**：品控主管的"快速放行"权限独立于审批流程，不参与正常分级审批
- **单一角色假设**：当前设计假设一人一角色，不允许多角色兼任

多租户场景适配：
当前系统为单租户设计。如需支持多租户，需在 User 表增加 `tenant_id` 字段，并在所有查询中附加 `WHERE tenant_id = ?` 条件；表结构层面的改动较小，主要是查询逻辑的调整。

## ⑥ V2 数据同步频率与一致性策略

| 策略项 | 选择 | 理由 |
|--------|------|------|
| 同步时机 | 每次上报异常时实时拉取 + 首次扫描时拉取 | 降低频繁轮询对 V2 系统的压力，同时保证关键操作（上报/扫描）时数据最新 |
| 本地快照用途 | 工单详情展示、离线降级展示 | 快照数据带时间戳标注"同步于 XX 时间" |
| V2 数据变更感知 | 实时校验策略：每次上报/扫描时调用 V2 接口校验运单存在性和金额 | 关键操作时实时验证，避免基于过期快照产生决策错误 |
| 不一致处理 | 记录差异由人工介入 | 定期对账或实时校验发现差异时记录到日志，人工判断处理 |

数据新鲜度在前端有明确标注：
- 实时获取的数据标注为"实时获取自 V2"
- 缓存数据标注为"本地缓存，同步于 [时间]"

## ⑦ 品控暂扣超时时长

| 参数 | 设定值 | 依据 |
|------|--------|------|
| 品控暂扣超时时长 | 30 分钟 | 品控暂扣涉及货物压仓，会产生仓储成本和影响后续发货排期。30 分钟是合理的处理窗口——既给品控主管留出复核时间，又不会因为长时间压仓影响仓库周转效率 |

**为什么独立于审批超时**：品控暂扣超时和审批超时是两套完全不同的业务逻辑：
- 品控暂扣超时的后果是"货物压仓产生运营成本"，驱动因素是仓储效率和货物周转，所以超时时长应远短于审批超时
- 审批超时的后果是"工单处理延迟"，驱动因素是人力和事务排期，所以超时时长可以相对宽松
- 品控暂扣超时后强制进入二级审批，加速问题升级处理

## ⑧ 品控规则触发阈值

| 规则 | 触发条件 | 依据 |
|------|---------|------|
| 数量差异 | 差异百分比 > 5% | 行业通用偏差容差为 3-5%，超过 5% 认为存在异常需要处理 |
| 外观破损 | 破损严重度 ≥ 2 级（1-5 级制） | 1 级（轻微划痕）可接受，2 级及以上（凹痕、破损）需触发异常 |
| 规格不符 | 偏差 > 10mm | 基于常见物流包装规格标准 |
| 标签错误 | 条码扫描不匹配 | 标签错误是严重问题，触发即视为异常 |
| 批次异常 | 过期日期检查失败 | 批次过期直接影响货物可用性，必须拦截 |

所有品控规则实现为 `QcRule` 表中的可配置项，支持后台动态调整，不允许硬编码。

## ⑨ 品控主管角色权限边界

| 权限项 | 是否拥有 | 说明 |
|--------|---------|------|
| 误判快速放行 | 是 | 认定扫描结果为误判时，可以一键解锁批次、关闭工单，无需走审批流程 |
| 查看品控工单 | 是 | 查看所有品控来源的异常工单 |
| 配置品控规则 | 是 | 调整品控触发阈值、异常子类型等规则参数 |
| 参与正常审批 | 否 | 不参与一级/二级分级审批流程 |
| 操作用户权限 | 否 | 不能创建/禁用用户 |
| 操作非品控工单 | 否 | 不能操作物流异常类工单的审批 |

**权限重叠说明**：品控主管与审批人的权限没有重叠。品控主管的"快速放行"是一个独立的操作路径，不经过分级审批引擎；审批人的审批权限也仅限于分级审批流程，不能做快速放行操作。

**一人多角色假设**：当前设计不支持一人兼任多角色。如需支持，需要在系统中增加角色组合判断逻辑，例如"某人同时是品控主管和二级审批人"时，需要明确其在某条工单上是以哪个角色操作。

## 可向产品经理提出的问题清单

如果允许向产品经理提问，以下是我的问题：

1. **金额阈值的业务来源**：分级审批的金额阈值是否有参考现有理赔数据的分布？还是需要从零确定？
2. **品控规则的判定依赖**：品控规则引擎是否需要接入外部检测设备（如电子称重、条码扫描枪）的自动输入？还是目前都靠人工判定后录入？
3. **多仓库/多商户**：当前系统是否立刻需要支持多仓库/多商户？还是可以按单租户设计、后续再改造？
4. **通知渠道**：审批超时/工单分配是否需要集成钉钉/飞书/企业微信通知？还是仅在系统内提醒？
5. **库存数据的归属**：库存数据是否完全由 V3 系统管理，还是通过接口调用 WMS 系统的库存数据？
6. **审批人离职兜底的具体流程**：审批人离职后，待审批工单是自动转交给同组其他人，还是需要管理员手动分配？
7. **品控暂扣超时后的具体流程**：超时后强制升级二级审批，是否需要通知相关审批人？

## 反思题简答

### 1. 如果题目直接给出了精确阈值和超时时长，架构设计会有什么不同？
在架构层面没有本质不同，因为阈值和时长本来就设计为可配置项。变化在于：假设说明文档的工作量减少；配置表中的默认值直接使用题目给定值，不再需要自行设定依据。

### 2. "审批状态变更"与"库存/赔付联动"之间一致性的最大风险点？
最大风险点是：审批状态已更新为"执行中"（executing），但下游的库存变动或赔付记录创建失败，导致出现"审批通过了但货物没出库/赔付没生成"的中间态。如果不使用数据库事务，可以改用"事件驱动+补偿任务"模式：审批通过后先写入事件表，由后台消费者异步处理联动操作，处理失败时写入重试队列并触发告警。

### 3. 面对一句话级别的模糊需求，你的倾向？
倾向于"先做出一个最小可用版本再迭代"。因为：在信息不完整时做过度设计可能方向全错；通过快速交付一个可用版本，可以在与产品经理的对焦过程中更快地暴露盲区、校准理解；但关键的安全/一致性相关假设必须提前确认，不能等上线了才发现设计失误。

### 4. 如果新增第六种异常类型"客户投诉服务态度"？
- 表结构：当前 `exception_ticket.exceptionType` 字段是 varchar 类型，直接新增一个枚举值（`"service_complaint"`）即可，不涉及表结构变更
- 状态机：无需改动，这类异常同样走分级审批流程
- 执行联动：需要定义新的下游动作映射。由于不涉及货品和库存，联动逻辑可能只包括"客服回访记录"和"处罚工单"，需要新增一个下游动作类型
- 改动成本：整体较小，主要是配置层面的新增，不需要改核心状态机或表结构

### 5. 工单量从 200 增长到 20 万条时，最先撑不住的环节？
- 最先撑不住的：**定时任务全表扫描**。当前超时检查通过全表扫描所有 pending/approval 状态的工单来判断是否超时，20 万条数据时每次扫描的开销和锁竞争会显著增大
- 解决方案：使用数据库索引加速（按 `status + updatedAt` 建复合索引），或改用每批次只检查最近 N 条的方式分片处理，或使用消息队列/定时调度来分批触发
- 其次：**列表查询的分页**。需要确保 status/exceptionType 等筛选条件有对应的索引，避免大数据量下的排序和分页性能问题

### 6. 给 V2 新增对外接口时如何保证不破坏现有调用方？
- **接口版本策略**：URL 路径中带上版本号（如 `/api/v2/waybills/v3/list`），新接口不改动原有端点
- **字段向后兼容**：新增接口字段时只做追加不做删除，老客户端调用时忽略未知字段即可
- **灰度上线**：先对 V3 系统的某个测试实例开放新接口，验证稳定后再全量开放；V2 接口升级时，V3 通过 Response Schema 校验未知字段来感知变更，记录日志告警
- **字段类型变更处理**：V2 的 `amount` 从 `int` 改为 `decimal` 时，V3 的 JSON 解析器需要兼容两种格式，并在解析后统一转为 `number` 类型；同时在同步日志中记录"字段类型变更"的警告
